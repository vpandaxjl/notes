00 — 问题背景

LLM 推理为什么慢？

大语言模型的推理速度受到自回归解码方式的根本性限制。

核心瓶颈 现代 LLM（如 Llama、Qwen）采用 Transformer 架构，每次前向传播只生成 1 个 token。生成 N 个 token 需要 N 次完整的大模型前向传播。GPU 算力无法被充分利用——因为每一步都依赖上一步的输出。
瓶颈 · 01

串行依赖

Token t 的生成依赖 token t-1，无法并行化生成多个 token。

The → quick → brown → fox → ??? → ???
瓶颈 · 02

Memory Wall（内存墙）

每次前向传播需从 HBM（GPU 显存）读取全部模型权重，带宽成为瓶颈而非算力。

读取延迟 ∝ 模型参数量 × 精度字节数 / HBM 带宽
# 70B 模型 @ BF16 ≈ 140GB，每步需完整读取
瓶颈 · 03

KV Cache 增长

随着序列变长，KV Cache 持续增大，每步注意力计算量线性增加。

KV Cache 大小 = 2 × L × H × d × seq_len × 精度
# L=层数, H=头数, d=head_dim；长文本推理成本极高
01 — 经典方法

自回归解码
Autoregressive Decoding

最基础的 LLM 推理方式：每步生成一个 token，每步调用一次完整的大模型。

步骤 · 01

输入 Prompt，计算初始 KV Cache

将输入序列一次性通过大模型（Prefill 阶段），得到所有输入 token 的 KV Cache，并生成第一个输出 token。

→

步骤 · 02

Decode 阶段：循环逐个生成

每轮将上一步生成的 token 作为新输入，模型只处理这一个 token（利用 KV Cache 避免重复计算），输出下一个 token。

DECODE 循环 — 每行 = 一次大模型前向传播
Step 1
Once → LLM → upon
1× forward
Step 2
upon → LLM → a
1× forward
Step 3
a → LLM → time
1× forward
Step N
time → LLM → ...
1× forward
关键代价 生成 N 个 token → 调用大模型 N 次 decode forward。GPU 大量时间浪费在等待 HBM 读写，而非真正计算。
步骤 · 03

复杂度分析

自回归解码的时间复杂度与输出序列长度成正比，无法利用 GPU 的大规模并行能力。

总时延 T_ar = N × T_forward(1 token)
# N = 目标输出长度
# T_forward(1) ≈ 模型大小决定的常数，通常数十毫秒
输出 100 tokens → 100 次大模型调用 GPU 利用率极低
02 — 传统改进

投机解码
Speculative Decoding

用小草稿模型快速生成候选序列，用大目标模型一次性并行验证，"以投机换速度"。

核心洞察 大模型验证多个 token 并不比验证 1 个 token 慢多少（KV Cache 可并行处理）。如果能提前猜对未来 k 个 token，就能以 1 次大模型 forward 的代价获得 k 个 token。
步骤 · 01

草稿阶段：小模型快速生成候选序列

用参数量小得多的草稿模型（Draft Model），自回归生成接下来 k 个候选 token。草稿模型很快，但质量低于大模型。

对比草稿模型与大模型在每个位置的概率分布，使用接受-拒绝采样决定是否接受每个草稿 token。第一个被拒绝的 token 之后的所有 token 一并丢弃。

步骤 · 04

传统投机解码的瓶颈

草稿模型本身也是自回归的，生成 k 个草稿 token 仍需 k 步串行推理，这部分延迟不可忽视。

总时延 T_spec = n/α × T_LLM + n/α × k × T_draft
# α = 平均每次 forward 被接受的 token 数
# k × T_draft = 草稿生成开销（串行！）
# 当 k 大时草稿开销不可忽略
草稿生成仍是串行瓶颈 k 较大时草稿延迟显著
03 — DFlash

Block Diffusion
扩散模型并行草稿生成

DFlash 的核心创新：用扩散模型替代自回归草稿模型，实现草稿 token 块内并行生成，彻底消除草稿阶段的串行瓶颈。

关键区别 传统投机解码的草稿模型是自回归的（串行生成 k 步）。DFlash 使用 Block Diffusion 模型，整个 k-token 块通过多步去噪并行生成——无串行依赖。
步骤 · 01

初始化：全块 MASK

将待生成的 k 个位置全部初始化为 [MASK] token（噪声状态），准备开始去噪过程。

06 — 发展脉络

推理加速技术演进

从朴素自回归到 DFlash 的技术发展路径。

自回归解码 (Transformer 基线)

每次 forward 生成 1 个 token，简单可靠，但无法并行化输出序列生成，GPU 算力严重浪费。

KV Cache 优化 (2022)

缓存历史 Key/Value，避免重复计算。Decode 阶段只需处理新增 token，大幅降低每步计算量，但未解决串行问题。

FlashAttention (Tri Dao, 2022)

IO感知的 Attention 实现，通过 Tiling 减少 HBM 读写，大幅提升硬件利用率。这是访存优化，不改变解码逻辑。

投机解码 (Leviathan et al., 2023)

引入草稿模型，将单次大模型 forward 的产出从 1 token 提升到 k tokens，理论上可达 k× 加速。受限于草稿质量和草稿生成串行开销。

DFlash (2026)

将扩散模型引入草稿生成阶段，实现 token 块内并行去噪。草稿生成开销与块大小 k 解耦，可使用更大的 k 值获得更高加速比，同时保持输出分布与原始大模型等价。
